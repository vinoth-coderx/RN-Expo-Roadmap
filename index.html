Module 01
Foundations
What is React Native
React Native is an open-source mobile framework by Meta that lets you build native Android and iOS apps using JavaScript and React. Unlike hybrid webview-based frameworks, RN renders real native UI components via a bridge to the platform's rendering engine.
Key Concept
React Native follows a "learn once, write anywhere" philosophy ‚Äî you share logic across platforms while keeping platform-specific UI when needed.
How React Native Works Under the Hood
RN runs your JS in a JavaScript engine (Hermes or JSC). The new architecture uses JSI (JavaScript Interface) for synchronous, direct communication between JS and native ‚Äî replacing the old asynchronous JSON bridge. Fabric handles the UI layer, and TurboModules handle native API access.

TanStack Query (React Query)
Why TanStack Query?
TanStack Query (formerly React Query) is the industry-standard library for server state management. It separates server state (data from APIs) from client state (UI state like modals, forms). It provides automatic caching, background refetching, stale-while-revalidate, pagination, infinite scroll, and optimistic updates ‚Äî all out of the box.

Key Mental Model
Client state = UI state (Zustand / Redux) ‚Üí "the sidebar is open"
Server state = API data (TanStack Query) ‚Üí "list of users from the backend"
Use both together ‚Äî TanStack Query for API data, Zustand for UI state.

Step 2: Provider Setup & Configuration
Wrap your app with QueryClientProvider. The QueryClient holds all cache configuration, retry logic, and defaults.
jsxsrc/lib/queryClient.ts
import { QueryClient } from '@tanstack/react-query';

export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 5 * 60 * 1000,       // 5 min ‚Äî data is "fresh" for 5 min
      gcTime: 10 * 60 * 1000,        // 10 min ‚Äî garbage collect unused cache
      retry: 2,                       // retry failed requests 2 times
      retryDelay: (attempt) =>        // exponential backoff
        Math.min(1000 * 2 ** attempt, 30000),
      refetchOnWindowFocus: false,    // disable for RN (no "window")
      refetchOnReconnect: true,      // refetch when network reconnects
    },
    mutations: {
      retry: 1,
    },
  },
});

jsxsrc/hooks/useUsers.ts ‚Äî Custom Query Hook
import { useQuery } from '@tanstack/react-query';
import { fetchUsers, fetchUserById } from '../api/userApi';

// ‚úÖ Query key factory ‚Äî keeps keys consistent across the app
export const userKeys = {
  all:        ['users'] as const,
  lists:      () => [...userKeys.all, 'list'] as const,
  list:       (filters: object) => [...userKeys.lists(), filters] as const,
  details:    () => [...userKeys.all, 'detail'] as const,
  detail:     (id: string) => [...userKeys.details(), id] as const,
};

// ‚úÖ Hook: Fetch all users
export const useUsers = () => {
  return useQuery({
    queryKey: userKeys.lists(),
    queryFn: fetchUsers,
  });
};

// ‚úÖ Hook: Fetch single user by ID
export const useUser = (id: string) => {
  return useQuery({
    queryKey: userKeys.detail(id),
    queryFn: () => fetchUserById(id),
    enabled: !!id,  // only fetch when id exists
  });
};

Step 4: useMutation ‚Äî Create, Update, Delete
Mutations handle write operations (POST, PUT, DELETE). After a successful mutation, you invalidate related queries so they refetch with fresh data.
jsxsrc/hooks/useUserMutations.ts
import { useMutation, useQueryClient } from '@tanstack/react-query';
import { createUser, updateUser, deleteUser } from '../api/userApi';
import { userKeys } from './useUsers';

// ‚úÖ CREATE user
export const useCreateUser = () => {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: createUser,
    onSuccess: () => {
      // Invalidate the users list ‚Üí triggers automatic refetch
      queryClient.invalidateQueries({ queryKey: userKeys.lists() });
    },
  });
};

// ‚úÖ UPDATE user
export const useUpdateUser = () => {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: updateUser,
    onSuccess: (updatedUser) => {
      // Update the specific user in cache immediately
      queryClient.setQueryData(
        userKeys.detail(updatedUser.id),
        updatedUser
      );
      // Also invalidate the list
      queryClient.invalidateQueries({ queryKey: userKeys.lists() });
    },
  });
};

// ‚úÖ DELETE user
export const useDeleteUser = () => {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: deleteUser,
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: userKeys.all });
    },
  });
};

Cache Invalidation Strategies
jsxInvalidation patterns
const queryClient = useQueryClient();

// 1. Invalidate ALL user queries (lists + details)
queryClient.invalidateQueries({ queryKey: ['users'] });

// 2. Invalidate only user lists
queryClient.invalidateQueries({ queryKey: ['users', 'list'] });

// 3. Invalidate one specific user detail
queryClient.invalidateQueries({ queryKey: ['users', 'detail', 'user-123'] });

// 4. Directly SET cache data (no refetch)
queryClient.setQueryData(['users', 'detail', 'user-123'], updatedUser);

// 5. Remove from cache entirely
queryClient.removeQueries({ queryKey: ['users', 'detail', 'user-123'] });

// 6. Invalidate EVERYTHING (e.g., after logout)
queryClient.invalidateQueries();

Optimistic Updates
Update the UI before the server responds. If the mutation fails, rollback to the previous state. This makes apps feel instant.
jsxhooks/useOptimisticUpdateUser.ts
export const useOptimisticUpdateUser = () => {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: updateUser,

    // ‚úÖ BEFORE mutation ‚Äî optimistically update cache
    onMutate: async (newData) => {
      // Cancel in-flight queries to avoid overwriting our optimistic update
      await queryClient.cancelQueries({ queryKey: userKeys.detail(newData.id) });

      // Snapshot the previous value (for rollback)
      const previousUser = queryClient.getQueryData(
        userKeys.detail(newData.id)
      );

      // Optimistically update the cache
      queryClient.setQueryData(userKeys.detail(newData.id), (old) => ({
        ...old,
        ...newData,
      }));

      // Return context with the snapshot
      return { previousUser };
    },

    // ‚ùå ON ERROR ‚Äî rollback to snapshot
    onError: (err, newData, context) => {
      queryClient.setQueryData(
        userKeys.detail(newData.id),
        context.previousUser
      );
    },

    // ‚úÖ ALWAYS ‚Äî refetch after success or error
    onSettled: (data, error, variables) => {
      queryClient.invalidateQueries({
        queryKey: userKeys.detail(variables.id),
      });
    },
  });
};
Rollback Pattern
onMutate ‚Üí snapshot + optimistic update ‚Üí onError ‚Üí rollback to snapshot ‚Üí onSettled ‚Üí refetch to sync. This guarantees UI is always consistent with the server.

SECTION ‚ë¶ Sync with Backend Server
WatermelonDB has a built-in sync protocol. It sends local changes to the server and pulls remote changes ‚Äî handling conflicts automatically.
jsxsrc/db/sync.ts
import { synchronize } from '@nozbe/watermelondb/sync';
import database from './index';

export const syncDatabase = async () => {
  await synchronize({
    database,

    // PULL ‚Äî get changes FROM server
    pullChanges: async ({ lastPulledAt }) => {
      const res = await fetch(
        `https://api.example.com/sync?last=${lastPulledAt}`
      );
      const { changes, timestamp } = await res.json();
      return { changes, timestamp };
      // changes = { posts: { created:[], updated:[], deleted:[] }, ... }
    },

    // PUSH ‚Äî send local changes TO server
    pushChanges: async ({ changes, lastPulledAt }) => {
      await fetch('https://api.example.com/sync', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ changes, lastPulledAt }),
      });
    },

    migrationsEnabledAtVersion: 1,
  });
};
Sync Protocol
Pull: Server returns { created, updated, deleted } per table since lastPulledAt.
Push: Client sends local creates/updates/deletes to server.
Use markAsDeleted() for soft-delete (sync-safe), destroyPermanently() only after sync.

Architecture Shift
Old Architecture vs New Architecture
The single biggest change in React Native's history ‚Äî the transition from the asynchronous Bridge to the synchronous JSI-based New Architecture ‚Äî completed in 2025. Here's what changed:
‚ùå Old Architecture (Pre-0.82)
Bridge-Based System
Async JSON Bridge between JS ‚Üî Native
All native modules loaded at startup
UIManager for UI rendering (slow)
JavaScriptCore engine (no AOT compile)
No concurrent rendering support
Manual forwardRef for component refs
Objective-C AppDelegate template
‚úÖ New Architecture (0.82+)
JSI-Based System
JSI: Synchronous C++ bridge ‚Äî zero serialization
TurboModules: Lazy-loaded native modules
Fabric: Synchronous rendering pipeline
Hermes V1: Improved bytecode + AOT compilation
Concurrent rendering, Suspense, auto-batching
Ref as props ‚Äî no more forwardRef needed
Swift AppDelegate template (iOS)
Performance Impact
Meta reported ~20-30% memory reduction and up to 39% faster rendering on component-heavy screens after migrating to the New Architecture. React Native 0.82+ made it mandatory ‚Äî legacy bridge flags are now ignored.

import { Activity } from 'react';

// <Activity> lets you hide UI without unmounting (preserves state!)
const TabScreen = ({ activeTab }) => (
  <View>
    {/* Home tab stays mounted but hidden ‚Äî keeps scroll position, state */}
    <Activity mode={activeTab === 'home' ? 'visible' : 'hidden'}>
      <HomeScreen />
    </Activity>

    <Activity mode={activeTab === 'search' ? 'visible' : 'hidden'}>
      <SearchScreen /> {/* search text preserved when hidden */}
    </Activity>
  </View>
);
jsxReact 19.2: useEffectEvent hook
import { useEffect, useEffectEvent } from 'react';

const ChatRoom = ({ roomId, theme }) => {
  // useEffectEvent ‚Äî reads latest values without re-triggering effect
  const onConnected = useEffectEvent(() => {
    showNotification(`Connected to ${roomId}`, theme); // always latest theme
  });

  useEffect(() => {
    const conn = createConnection(roomId);
    conn.on('connected', onConnected);
    conn.connect();
    return () => conn.disconnect();
  }, [roomId]); // ‚Üê 'theme' NOT in deps, but onConnected reads latest
};
